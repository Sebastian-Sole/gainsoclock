# 1.2.0 post-release: Convex migrations + History sync fixes

Context for the 1.2.0 release (build 5, `e2bce59`). Two things live here: the
manual Convex commands that had to be run against **prod** after the release,
and the two History bugs that surfaced afterward with their diagnosis + fixes.

## Manual Convex commands run after 1.2.0

The release tooling (`pnpm release:*`, `scripts/bump-version.mjs`) only bumps
version/build numbers. These migrations/backfills are **separate, manual**, and
were run by hand against prod. All three shipped in 1.2.0 and are one-shot.

```bash
# 1. Exercise-metrics migration (composable metrics, #97) — dry run then commit
npx convex run --prod migrations:runMigrateExerciseMetrics '{"logOnly": true}'
npx convex run --prod migrations:runMigrateExerciseMetrics '{"logOnly": false}'

# 2. Interval-sets migration (single-set intervals) — dry run then commit
npx convex run --prod migrations:runMigrateIntervalSets '{"logOnly": true}'
npx convex run --prod migrations:runMigrateIntervalSets '{"logOnly": false}'

# 3. Watch↔log backfill (#117) — PAGINATED: loop feeding back continueCursor
#    until the result returns isDone: true.
npx convex run healthData:backfillAllWorkoutLinks '{}' --prod
npx convex run healthData:backfillAllWorkoutLinks '{"usersCursor":"<continueCursor>"}' --prod
```

Notes:
- Run the metrics migration **before** the interval-sets one (exercises must be
  migrated first so join rows read a populated parent `metrics`).
- Use `pnpm dlx convex ...` (or a pnpm script) rather than `npx` to avoid the
  `Unknown project config "node-linker"` npm warning — this repo is pnpm-only.

## Post-1.2.0 History bugs

Two symptoms reported in prod after 1.2.0:
1. **Stats → History shows "0 exercises"** on imported workouts (not on
   workouts logged in-app).
2. **Apple Watch recordings don't link** to the Fitbull log for the same
   session (#117 was supposed to handle this).

### Diagnosis

Diagnosed with a throwaway read-only Convex query (`inspectUserLinks`, since
removed) that resolved a user by email and reported:
- link coverage over `externalWorkouts` (+ re-runs the matcher to see how many
  unlinked ones *would* match a log right now), and
- whether the server actually holds exercise rows for the user's logs.

Findings for the affected account (1009 logs):
- `workoutLogs.total: 1009`, `totalExerciseRows: 4411`, `logsWithZeroExercises: 0`
  → **server data is fully intact.** Bug 1 is purely client-side hydration.
- `externalWorkouts: linked 0, unlinked 32, unlinkedButWouldMatch 0`
  → the matcher and link-writing work; **not one watch workout time-overlaps
  any log.** Strongly suspected cause: FitNotes-imported logs carry synthetic
  timestamps (round-hour, e.g. `10:00:00.000Z`) that can't overlap a real
  Apple Watch recording. (Confirm with `nearestLogSamples`.)

### Bug 1 root cause + fix (DONE)

`listMeta` (the query that keeps the History store fresh) returns only
session-level fields — **no exercises** (the `workoutLogs` table has none; they
live in `workoutLogExercises`/`workoutSets`). The one-shot `listFull` seed only
runs when the local store is *empty* (`convex-sync-provider.tsx`). So any log
that arrives via `listMeta` without ever being a full local copy on this device
(imported #108 / cross-device) renders "0 exercises". Overview stats still work
because they only use session-level fields.

Fix: **lazy per-log hydration** (chosen over an unbounded `listFull` re-seed,
which is unsafe near Convex's ~16k-doc read limit at 1000+ logs):
- `convex/workoutLogs.ts` → `getLogDetail({ clientId })` — one log's detail,
  reads bounded to that log.
- `stores/history-store.ts` → `hydrateLogDetail(serverLog)` patches only the
  matching row (module-level `serverLogToLocal` mapper extracted for reuse).
- `components/history/workout-log-card.tsx` → a card with 0 exercises fetches
  its detail and patches it in; skips once populated.

### Bug 2 — confirmed a data limitation, not a code bug

The reverse-matcher already exists and is wired into both `workoutLogs.create`
(`:296`) and `update` (`:524`); forward path is `healthData.upsertExternalWorkouts`.
`nearestLogSamples` settled it: every nearest log sits at `22:00–23:00Z` with a
1-hour window — i.e. **local midnight in CEST (UTC+2), a synthetic 1-hour
duration**. FitNotes CSV carried only a date, so the importer stamped each log
at local midnight. The watch↔log start gaps are scattered (104–1302 min), not a
constant offset, which rules out a timezone bug in the matcher. `unlinkedButWouldMatch: 0`:
no change to the existing time-overlap matcher can link these — there is no real
time to overlap against.

Scope: most of the 32 externals are running/hockey — cardio never in FitNotes
(a lifting logger), so they *correctly* show standalone. Only same-day
strength (watch + FitNotes import) is the real "should merge" case, and it
can't auto-link because of the synthetic timestamp.

### Bug 2 — solution shipped (manual + suggested merge)

Decided against a push-per-import notification (fires redundantly for live
logging, storms on backlogs, can't one-tap merge with >1 candidate). Built a
confidence-tiered merge instead:

- **Auto** — real time-overlap, already live (`convex/workoutOverlap.ts`).
- **Suggest** — a same-day, activity-compatible log surfaces a one-tap "Merge"
  chip on the imported card (`lib/merge-suggestions.ts` + `lib/workout-activity.ts`,
  both unit-tested). Gated so a running watch workout is never suggested for a
  strength log.
- **Manual** — expand any imported card to pick which same-day log it belongs
  to, or "Keep separate". New `healthData.linkExternalWorkout` mutation (inverse
  of `unlinkExternalWorkout`); no time overlap required.
- **Batch review** — a "Review imported workouts" banner in History opens a
  sheet to resolve the whole backlog in one pass (`merge-review-modal.tsx`).

Files: `lib/workout-activity.ts`, `lib/merge-suggestions.ts` (+ tests),
`convex/healthData.ts` (`linkExternalWorkout`), `components/history/external-workout-card.tsx`,
`components/history/merge-review-modal.tsx`, `components/stats/history-tab.tsx`.

Still worth checking: does the FitNotes export carry a **time** column we're not
parsing? If so, fixing the importer would let many of these auto-link and shrink
the backlog the merge UI has to handle.

### Re-diagnosing later

The diagnostic query was a one-off and was removed after the fix shipped. To
triage a similar report again, a small read-only internal query that (per user)
counts `externalWorkouts` link coverage, re-runs `bestMatchingLog` from
`convex/workoutOverlap.ts` against same-window logs, and counts
`workoutLogExercises` rows per log is enough to split "client hydration" from
"server data loss" from "no time overlap".
