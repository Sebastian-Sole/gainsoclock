# Security Review v2 — Onboarding Flow Master Plan (revised)

**Reviewer:** security persona
**Date:** 2026-04-21 (v2)
**Verdict:** APPROVED

## Summary

v2 resolves every blocking critical and every listed concern from the v1 review, in several cases going beyond what v1 asked for (append-only consent log with explicit index, in-memory-only Art. 9 slice with partitioned AsyncStorage, three-way unbundled consent purposes including analytics-default-off, dual-token webhook window with timing-safe compare, route-allowlisted session replay with `maskAllInputs` global, and moderation-pass on LLM output). One intentional deferral — in-app Art. 20 export → V1.1 — is handled correctly in the changelog by adjusting the S6 copy to not promise in-app export, which is the right failure mode for Art. 7(3) / EDPB 05/2020 compliance. Verdict flips from v1 CHANGES NEEDED → APPROVED.

## v1 Concerns — Now Resolved

- **CR1 (HealthKit firewall inert).** §3.3 now uses the correct distributive conditional `Extract<keyof T, ForbiddenKeys> extends never ? T : never`, applies it at the call site via `event: E & { props: NoHealthKitFields<E["props"]> }`, adds a runtime key-scan (throw in `__DEV__`, drop+warn in prod), extends `ForbiddenKeys` to cover derived metrics (`tdee`, `bmr`, `bmi`, `caloriesBurned`, etc.), and adds a negative type-test file `lib/analytics.test-types.ts` as a Phase 3 exit criterion. This is a stronger fix than v1 asked for.
- **CR2 (Art. 9 pre-consent in AsyncStorage).** §3.8 keeps special-category fields (age, weight, height, body fat, biological sex) in-memory only; only S2–S4 non-special fields persist, partitioned by userId. Explicit purge rules on mutation `onSuccess`, sign-out (wired into `auth-cache-store.clear()`), user reset, and cold-boot partition mismatch. Debounced writes (300ms + persist-on-blur) per Performance #4.
- **CR3 (Withdrawal / Art. 17 / Art. 20 deferral).** §3.10 ships withdrawal UI (`app/settings/privacy.tsx`) and account deletion (`app/settings/delete-account.tsx`) in V1 (Phase 8). `withdrawConsent` appends a new row (never mutates). Cascades documented: `ai_coach_inference` withdrawal archives `onboardingAha` rows, `health_data_personalization` schedules profile erasure, `analytics` calls PostHog server-side delete API. Art. 20 is explicitly deferred to V1.1 with matching S6 copy ("withdraw in Settings") — no false promise. Account-deletion cascade enumerates all owned tables plus HealthKit `HKExternalUUID` cleanup and PostHog delete — meets Apple 5.1.1(v) and GDPR Art. 17.
- **CR4 (Consent idempotency underspecified).** §3.1 `userConsents` is now an append-only log with `by_user_purpose_grantedAt` index; §3.4 documents `getConsents()` as latest-per-purpose reduction; `withdrawConsent` appends `granted: false`. Server-authored `grantedAt`. Matches the "at-least-once writes, append-only log" pattern regulators prefer.
- **CR5 (SIWA ↔ email collision).** §2 S1 adds a collision branch in `convex/auth.ts` — if a SIWA `sub` has no existing user but a same-email row exists, surface a support-path instead of creating a silent double-account. Relay `@privaterelay.appleid.com` is documented as authoritative identity. Phase 4 includes pre-flight research on `@convex-dev/auth@0.0.90` account-linking, and `usesAppleSignIn: true` + Apple Developer entitlement is a pre-TestFlight gate.
- **#6 (OpenAI refusal + partial-JSON parse).** §3.5 explicitly prohibits `JSON.parse` on partial stream buffer — client renders skeleton until `status: "complete"`. Refusal shape handled as failure → safety-net session. Post-parse validator rejects out-of-tier durations and `sets*reps` over caps. Prompt injection mitigated by literal-union goal validator + profile passed as JSON user-message (not prompt interpolation) — better than v1 asked.
- **#7 (posthog-node shutdown race + env assertion).** §3.3 wraps `client.shutdown()` in `Promise.race([..., timeout(2000)])`. §3.5 asserts `process.env.OPENAI_API_KEY` at action entry with typed error. Explicit "never log the key" note.
- **#8 (RC webhook replay tests + token rotation).** Phase 2 exit criteria now require replay-tests: old-timestamp `INITIAL_PURCHASE` no-ops; `SUBSCRIPTION_EXTENDED` with past timestamp still updates `trialExpiresAt`. Dual-token window `REVENUECAT_WEBHOOK_AUTH_TOKEN` + `*_PREVIOUS` with 7d overlap, timing-safe compare, `docs/revenuecat-webhook-rotation.md`. Exceeds v1 ask.
- **#9 (Session replay + Privacy Label).** §3.3 sets `maskAllInputs: true` globally, denies replay on all `(auth)/*` routes, S5/S5a/S5b primer/prefill, S7/S8/S11. Privacy Label (§2 S0) now declares Contact Info → Email, Identifiers, User Content. Replay buffer capped at 5MB.
- **Obs #1 (`getAuthUserId` discipline).** §3.4 explicitly restates the discipline for every new module (`onboarding`, `onboardingActions`, `analytics`, `home`). `userId` never a public arg; `internalMutation` is the only surface that may take `userId`. Phase 1 exit criterion.
- **Obs #2 (OAuth redirect URIs) / SIWA entitlement.** Phase 4 lists `usesAppleSignIn: true` + Apple Developer entitlement as a pre-ship gate.
- **Change #5 (literal-union goal validator).** Done — `goalValidator` in `convex/validators.ts`, both `goals[]` and `primaryGoal` reference it, array cap of 4 enforced server-side in the mutation body.

## v1 Concerns — Still Open

None. All v1 criticals and concerns are resolved in v2.

## New Concerns Introduced by v2 Revisions

- **Moderation-pass failure mode on `intro` (§3.5).** The plan passes the generated `intro` through `openai.moderations.create()` and logs flags to an `aiSafetyIncidents` table. Two residual gaps worth noting (non-blocking): (a) the moderations API call itself is an external HTTP round-trip inside the action — if it times out, the plan doesn't say whether to fail-open (publish the intro) or fail-closed (serve static). The safer posture is fail-closed to the static intro ("Here's your first session — let's start."); add one line to §3.5 post-parse section. (b) `aiSafetyIncidents` should be subject to the same Art. 17 deletion cascade as other owned tables — §3.10 already enumerates it in the cascade list, which is correct. No change required; flagging for implementation review.
- **Resend as a new sub-processor (§3.2 + methodology page).** v2 introduces Resend as the email provider for the 48h reminder and DCSA notifications. §2 S6 lists sub-processors as "OpenAI US/SCC, PostHog EU-Frankfurt, RevenueCat US/SCC, Convex per region declaration" on the methodology page — **Resend is missing from this list**. Resend operates in both US and EU regions; the plan specifies EU region wiring, so the sub-processor disclosure should add "Resend (EU region, transactional email)" to the methodology page and the Privacy Nutrition Label discussion. This is a documentation fix, not a code change — low severity, but should be caught before ship.
- **`onboardingAha.profileSnapshot` contains Art. 9 data (§3.1).** The schema stores `profileSnapshot: v.string()` — a JSON snapshot of the user profile for re-generate comparison. This snapshot will contain weight/height/age/body-fat. Two points: (a) it correctly falls under `ai_coach_inference` consent and cascades on `deleteAccount()` since the table is enumerated in §3.10. (b) Consider documenting a retention policy — if the user completes onboarding and never re-generates, the snapshot persists for the user lifetime. A reasonable posture is "purge `profileSnapshot` 30 days after last regenerate, retain the `workout` output." This is V1.1 polish, not a v1 blocker.
- **`openai-config.ts` model names exposed via env vars (§3.5).** `OPENAI_AHA_MODEL` / `OPENAI_AHA_FALLBACK_MODEL` read from Convex env. Non-secret, but misconfiguration (e.g. setting to a deprecated model) would fail open. Phase 10 env-var enumeration table covers this; assertion at action entry plus the fallback retry makes this low risk.

## What's Handled Exceptionally Well

- **Three-way unbundled consent with analytics-default-off and pre-consent buffering.** §3.3's "hold `intake_started` in in-memory buffer from S1, flush only if analytics granted at S6, else drop" is the correct Datatilsynet-enforcement posture and is rare to see spelled out this carefully. Exceeds what v1 asked for.
- **Append-only consent log with explicit `by_user_purpose_grantedAt` index.** Clean audit trail, replay-safe, withdrawal semantics obvious.
- **Profile passed as JSON user-message, not prompt-interpolated.** §3.5's prompt-injection mitigation is structurally robust — a user typing "ignore previous instructions" into their goal literally cannot reach the system prompt because the validator rejects non-enum values, and the profile is passed in its own message frame.
- **HealthKit firewall is now genuinely enforceable.** Type + runtime + negative type test is defense-in-depth that will survive a future refactor that widens a `props` type.
- **Analytics consent gate threaded through `capture()`.** Not a type-level trick — the function checks `userConsents` at call time. Matches Art. 6(1)(a) + Art. 9(2)(a) posture.
- **Out-of-order protection + timing-safe compare on the RC webhook.** Production-grade.
- **Server-authored timestamps across the board.** `grantedAt`, `createdAt`, `updatedAt` are never client-supplied — matches the client-trust posture the code conventions require.

## Final Verdict Rationale

APPROVED. Every v1 critical and concern has a concrete, cited fix in the revised plan, and several resolutions exceed the minimum v1 asked for (append-only log with index, three-purpose consent instead of two, runtime + type HealthKit firewall, dual-token webhook window with replay tests). The three new concerns I surfaced are documentation-level (Resend sub-processor disclosure, moderation-pass fail-closed posture, `profileSnapshot` retention) and none rise to blocking. The plan's security posture for onboarding — Art. 9 consent sequencing, SIWA identity handling, webhook auth hardening, prompt-injection surface area, and Art. 17/withdrawal cascade — is defensible against OWASP Top 10 2025 (A01, A02, A03, A04, A08) and against Datatilsynet / EDPB 05/2020 enforcement posture. Ship when Phase 8 exits.
